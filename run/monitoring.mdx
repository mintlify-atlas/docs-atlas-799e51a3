Reth exposes Prometheus-compatible metrics and supports OpenTelemetry trace export, giving you full visibility into sync progress, peer connections, database performance, and RPC latency.

## Enabling metrics

Start Reth with the `--metrics` flag, specifying the address and port to expose the metrics endpoint:

```bash
reth node --metrics 127.0.0.1:9001
```

To verify the endpoint is working, curl it directly:

```bash
curl 127.0.0.1:9001
```

The response is a text-format Prometheus metrics dump. To watch metrics update in real time from the terminal:

```bash
while true; do date; curl -s localhost:9001 | grep -Ev '^(#|$)' | sort; echo; sleep 10; done
```

## Setting up Prometheus and Grafana

For historical tracking and visualization, use Prometheus to scrape the metrics endpoint and Grafana to display them on a dashboard.

### Install Prometheus and Grafana

<Tabs>
  <Tab title="macOS">
    ```bash
    brew update
    brew install prometheus
    brew install grafana
    ```
  </Tab>
  <Tab title="Debian/Ubuntu">
    ```bash
    # Install Prometheus
    PROM_VERSION=$(curl -s https://api.github.com/repos/prometheus/prometheus/releases/latest | grep tag_name | cut -d '"' -f 4 | cut -c 2-)
    wget https://github.com/prometheus/prometheus/releases/download/v${PROM_VERSION}/prometheus-${PROM_VERSION}.linux-amd64.tar.gz
    tar xvfz prometheus-*.tar.gz
    cd prometheus-*

    # Install Grafana
    sudo apt-get install -y apt-transport-https software-properties-common
    wget -q -O - https://packages.grafana.com/gpg.key | sudo apt-key add -
    echo "deb https://packages.grafana.com stable main" | sudo tee -a /etc/apt/sources.list.d/grafana.list
    sudo apt-get update
    sudo apt-get install grafana
    ```
  </Tab>
  <Tab title="Fedora/RHEL/CentOS">
    ```bash
    # Install Prometheus
    PROM_VERSION=$(curl -s https://api.github.com/repos/prometheus/prometheus/releases/latest | grep tag_name | cut -d '"' -f 4 | cut -c 2-)
    wget https://github.com/prometheus/prometheus/releases/download/v${PROM_VERSION}/prometheus-${PROM_VERSION}.linux-amd64.tar.gz
    tar xvfz prometheus-*.tar.gz
    cd prometheus-*

    # Install Grafana
    sudo dnf install -y https://dl.grafana.com/oss/release/grafana-latest-1.x86_64.rpm
    ```
  </Tab>
  <Tab title="Windows">

    ```powershell

    choco install grafana

Extract Prometheus (e.g. to `C:\prometheus`) and install Grafana using the `.msi` installer.

### Configure Prometheus to scrape Reth

By default, Prometheus scrapes itself. You need to update its config to point at Reth's metrics endpoint at `localhost:9001`.

An example Prometheus config is available in the Reth repository at [`etc/prometheus/prometheus.yml`](https://github.com/paradigmxyz/reth/blob/main/etc/prometheus/prometheus.yml). Common config file locations:

- **macOS (Homebrew)**: `/opt/homebrew/etc/prometheus.yml`
- **Linuxbrew**: `/home/linuxbrew/.linuxbrew/etc/prometheus.yml`
- **Other Linux**: `/usr/local/etc/prometheus/prometheus.yml`

<Note>
  Port `9001` is where **Reth** exposes metrics for Prometheus to scrape. Port `9090` is where **Prometheus** exposes metrics for Grafana to consume. These are separate endpoints serving different purposes.
</Note>

### Configure Grafana

<Steps>
  <Step title="Open Grafana">
    Navigate to `http://localhost:3000` in your browser. The default credentials are `admin` / `admin`.
  </Step>
  <Step title="Add Prometheus as a data source">
    In the left sidebar, go to **Connections** → **Data Sources** → **Add data source**. Select **Prometheus** as the type and enter `http://localhost:9090` as the HTTP URL. Click **Save & Test**.
  </Step>
  <Step title="Import the Reth dashboard">
    Click the squares icon in the top-left, then **New** → **Import**. Click **Upload JSON file** and select the dashboard at [`reth/etc/grafana/dashboards/overview.json`](https://github.com/paradigmxyz/reth/blob/main/etc/grafana/dashboards/overview.json). Choose the Prometheus data source you just created and click **Import**.
  </Step>
</Steps>

Once imported, the dashboard displays sync stage progress, peer counts, database read/write latency, and more. The dashboard will appear empty until your node connects to peers and begins syncing.

## OpenTelemetry tracing (OTLP)

Reth can export spans and logs to any OpenTelemetry-compatible backend using the `--tracing-otlp` flag.

```bash
reth node --tracing-otlp=http://localhost:4318/v1/traces
```

Compatible backends include [Grafana Tempo](https://grafana.com/oss/tempo/) and [Jaeger](https://www.jaegertracing.io/).

### Running Jaeger locally

```bash
docker run --rm --name jaeger \
  -p 16686:16686 \
  -p 4317:4317 \
  -p 4318:4318 \
  -p 5778:5778 \
  -p 9411:9411 \
  cr.jaegertracing.io/jaegertracing/jaeger:2.11.0
```

Then start Reth pointing at the Jaeger OTLP collector:

```bash
reth node --tracing-otlp=http://localhost:4317 --tracing-otlp-protocol grpc
```

Traces are searchable and viewable at `http://localhost:16686`.

<Tip>
  For nodes processing a high volume of transactions or blocks, increase `OTEL_BLRP_MAX_QUEUE_SIZE` above its default of `2048`. If the queue fills up faster than the exporter can flush, spans and log records will be dropped silently.
</Tip>

## Key metrics to watch

<AccordionGroup>
  <Accordion title="Sync progress">
    Track which stage Reth is currently in and how far through the stage it has progressed. The `reth_stages_*` metrics report block numbers and execution counts per stage.
  </Accordion>
  <Accordion title="Peer connections">
    Monitor `reth_network_connected_peers` to ensure the node maintains a healthy number of peers. A low peer count can stall sync.
  </Accordion>
  <Accordion title="Database performance">
    Watch `reth_db_*` metrics for read/write latency and transaction counts. High latency may indicate disk I/O saturation.
  </Accordion>
  <Accordion title="RPC latency">
    `reth_rpc_*` metrics expose request counts and duration histograms per method. Useful for diagnosing slow or high-volume API access.
  </Accordion>
  <Accordion title="Memory and CPU">
    Use your system's standard process metrics (via node_exporter or cAdvisor) alongside Reth's application metrics for a complete picture of resource utilization.
  </Accordion>
</AccordionGroup>
